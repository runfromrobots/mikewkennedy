# Project: Michael Kennedy (MK) Portfolio Site

Personal portfolio/resume site for Michael Kennedy, a Senior UX Researcher (13+ years) targeting Senior/Lead-level UX Research roles, fully remote. Static site, no build step, no framework.

## Stack

Plain HTML/CSS/JS. No React, no bundler, no dependencies. Fonts loaded via Google Fonts CDN link tags in each HTML file's `<head>`.

## File structure

```
index.html                     homepage (hero, about, experience, toolkit, work, testimonials, contact)
styles.css                      shared stylesheet for the whole site
script.js                        shared JS (footer year, nav scroll-spy, carousels, site banner)
case-studies/index.html         case studies landing page (links back via ../)
how-this-site-was-made.html    meta write-up about building this site with Claude Code (see below), lives at root, not linked from main nav or the case studies grid
images/                         drop images here, reference as images/filename.jpg
.nojekyll                        present so GitHub Pages serves files as-is
DEPLOY.md                       deployment notes (Netlify/GitHub Pages/Vercel)
```

## Design system

* Colors: navy `#0A2540` (surface accent — buttons, contact section), electric cyan `#22E5FF`/`#33CFEA` (primary accent — headings, links, borders, replaces the old teal), magenta `#FF2EC4` (secondary accent, used sparingly for gradients/hover), near-black `#08090D` (background), see `:root` in styles.css for full token list. The `--teal`/`--teal-deep` variable names are historical (kept so far fewer call sites needed touching) but now hold cyan values, not teal.
* Type: Fraunces (display/headlines), Inter (body), IBM Plex Mono (labels, dates, eyebrows, tags)
* Hero graphic: the original animated "signal convergence" SVG (research sources converging into one insight node) was replaced with a two-photo collage — `images/ibm-research.jpg` (larger, cyan-bordered, "standing behind the glass at IBM") and `images/taiwan-team.jpg` (smaller, magenta-bordered, overlapping the first photo's corner, working with colleagues in Taiwan). `.hero-photos` / `.hero-photo-main` / `.hero-photo-accent` in styles.css. The CRT scan-sweep effect that used to cross the SVG now sweeps across this photo collage instead (`.hero-photos::after`). `images/hero-photo.jpg` (the alley/sunglasses photo that used to sit at the SVG's convergence point) is no longer used anywhere — kept in `images/` in case it gets placed elsewhere later.
* Keep all new sections consistent with this system rather than introducing new colors/fonts
* Spacing, small-text sizing, and breakpoints are systematized on the Carbon Design System (github.com/carbon-design-system) scale — CSS custom properties prefixed `--cds-` in `:root` (spacing-01 through 13, a handful of type-scale steps, a motion duration/easing pair). This is a token-level adoption only ("Option A"): Fraunces/Inter for display/body type and the signature SVG structure are unchanged — deliberately not a full swap to literal Carbon components/theme, which would clash with the site's personal-portfolio identity. (The brand *colors* were later overhauled separately for the hacker theme below — Option A was about spacing/grid/type-scale infrastructure, not colors.) Breakpoints follow Carbon's md (672px) and lg (1056px). When adding new spacing/gaps, reach for a `--cds-spacing-*` token instead of a new hardcoded px value.
* `--cds-caption-01` (nav, eyebrow, tags, dates, footer, all the mono/label text), `--cds-body-compact-01`/`--cds-body-01` (credentials list, a few other spots), and `--cds-body-02` (hero sub-copy, a few other body spots) were bumped up from Carbon's literal values over three rounds — 0.75rem to 1.125rem, 0.875rem to 1.1875rem, and 0.875rem to 1.125rem respectively — because the smallest text on the page kept reading too small/barely legible, especially set in VT323. `.log-company` (Experience section) is hardcoded rather than token-based and was bumped alongside these to 1.125rem so it stays at least as large as `.log-date`, which reads off `--cds-caption-01`. Since every one of those elements reads off these tokens, don't hardcode a smaller font-size on top of them for new label-style text — that would silently undo the legibility fix.
* Buttons: flat fill only, no outline/ghost/bordered-transparent style anywhere on the site. `.btn-primary` is a cyan-to-magenta gradient; `.btn-secondary` is a cyan-to-navy gradient (`.contact .btn-secondary` swaps this to magenta-to-navy so the contact pair reads as two distinct colors against the navy background) — both have dark text and a cyan glow at rest. **Hover is unified across every button**: solid `--accent-purple` (a subdued/desaturated purple) with light text, `.btn-primary:hover, .btn-secondary:hover` in one rule — deliberately not swapping to the other button's own color, which used to read as a mismatch when two different-colored buttons sat side by side. Every button uses the oversized `.btn-large` (or `.btn-header`) treatment per Fitt's Law — a shared `--action-btn-size` token (80px, Carbon's largest button size). The header "Get in touch" button lives outside `.wrap` as a direct sibling in `.site-header` (which is itself the flex container now) so it spans the full header bar height *and* reaches the true right edge of the page, not just the centered content column — `.header-content` wraps just the logo+nav so they still align with `.wrap`'s max-width. Paired buttons touch with zero gap (`.hero-actions`, `.contact-links` both `gap: 0`) so each pair reads as one continuous bar. All button color rules live in one place (base Buttons section + one `.contact` override block) — don't re-scatter overrides across a separate "effects layer" again, that's what caused the mismatch in the first place.
* **This site has one theme only: a permanent dark/terminal "hacker" theme** (explicitly requested — nod to 1995's *Hackers* and CRT/terminal computing). There is no light mode, no toggle, and no `prefers-color-scheme` switch — that was tried and deliberately removed. Don't reintroduce a light theme or make the palette conditional without being asked.
* VT323 (a pixelated CRT-terminal Google Font) is loaded and used via `--font-mono` for all label/eyebrow/mono text (nav, eyebrow, dates, tags, footer) — Fraunces/Inter still handle headings/body copy.
* Effects layer, near the end of styles.css, unconditional (not media-query-gated): cyan glow on h1/h2/h3 (`text-shadow`), blurred cyan/magenta gradient blobs behind the hero and behind `.contact` (`.hero::before/::after`, `.contact::before/::after` — same technique, same colors, deliberately matching), a slow CRT scan-sweep across the hero photo collage (`.hero-photos::after`, `@keyframes scan-sweep`), a fixed faint scanline texture over the whole page (`body::after`), and a one-shot (not looping) RGB-split glitch flicker on nav-link/`.link-cta` hover (`@keyframes glitch-flicker`). Keep new effects similarly restrained — a hover-triggered one-shot, not a constant loop, and check `prefers-reduced-motion` still disables animations (the existing reduced-motion rule at the top of styles.css covers this automatically via `* { animation-duration: 0.001ms !important; ... }`).
* Experience section (`#log` / `.experience`) is `--paper-raised` (a shade lighter than the page's `--paper` background) so it lifts slightly off the page, and every other `.log-entry` row gets a faint cyan tint (`nth-child(even)`) to zebra-stripe them — both subtle enough not to affect text contrast.
* Each `.log-when` block splits company name and date into separate `.log-company` / `.log-date` spans (not a single `<br>`-joined string). Both use IBM Plex Mono rather than the site's default VT323 label font — VT323 read too narrow/chunky for dates at small sizes. `.log-company` is upsized (1rem), medium weight, uppercase, letter-spaced, cyan, for visual prominence; `.log-date` stays small (`--cds-caption-01`), regular weight, `--ink-soft` colored, understated. This was chosen from three side-by-side treatments (an all-Fraunces editorial option and an Inter/Plex hybrid were the other two) — don't relitigate without being asked.
* The "MK" brand logo in the header is deliberately oversized (120px, vs. the ~80px header bar) and bleeds down past the header's bottom edge into the section below by design — `.brand` is `position: absolute` sized/positioned for this, while an invisible `.brand-spacer` (44px) holds its old footprint in the header's flex flow so the nav doesn't shift. Caveat: since `.site-header` is `position: sticky`, this bleed stays pinned during scroll and will overlap whatever section content is currently just below the header, not only the hero — flagged as a known/accepted side effect, not a bug, unless told otherwise.
* `--heading` is a separate token from `--navy` (cyan vs. navy) so headings/text use the accent color while navy surfaces (buttons, the contact section, the skip link) stay literally navy — when adding new navy-colored text (not a background/fill), use `var(--heading)`, not `var(--navy)` directly.
* `body { color-scheme: dark; }` is set so browser chrome (scrollbars, form controls) matches.

## Conventions

* No em dashes in any content or copy written for this project
* Avoid "it's not X, it's Y" contrasting constructions in copy
* Keep prose concise and direct, avoid over-explaining
* Relative links only (`case-studies/`, `../`, `images/...`) so the site works correctly whether served from a domain root or a GitHub Pages subpath
